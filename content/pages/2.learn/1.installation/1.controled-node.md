# Préparation du noeud de controle

Cette page permet de configurer le noeud de controle utilisé pour déployer TDP. Il pourra à la fois être utilisé pour un déploiement manuel ou avec [TDP Manager](#).

## Prérequis

Le seul prérequis pour configurer le noeud de controle est d'avoir accès à [Ansible](https://www.ansible.com/). Les systèmes d'exploitation suivants ont été testés :

- CentOS 7
- Rocky 8
- AlmaLinux 8

L'installation peut être effectuée sur d'autres systèmes d'exploitation, mais il est possible que certaines fonctionnalités ne soient pas disponibles.

Note : Dans le cadre de l'installation de TDP Manager, il est recommandé d'utiliser directement la version d'Ansible contenue dans l'[envionnement virtuel de TDP Manager](#).

## Configuration d'Ansible

Le fichier de configuration d'Ansible, `ansible.cfg`, demande certaines spécificités propre à TDP :

```ini
[defaults]
inventory=/path/to/tdp/inventory ; OBLIGATOIRE chemin vers le répertoire contenant l'inventaire Ansible.
collections_paths=/path/to/tdp/collections/ ; RECOMMANDE chemins vers les répertoires contenant les collections Ansible.
display_skipped_hosts=False ; RECOMMANDE permet de ne pas afficher les tâches Ansible qui sont _skipped_ pour éviter d'encombrer les logs.
any_errors_fatal=True ; OBLIGATOIRE permet d'arrêter l'exécution d'Ansible dès qu'il y a une erreur et empêche Ansible de continuer sur les hôtes restants.

; RECOMMANDE permet d'accélérer considérablement le lancement des playbooks, car l'installation de TDP implique de nombreux appels à la commande `ansible-playbook`.
[inventory]
cache = true
cache_plugin = jsonfile
cache_timeout = 7200
cache_connection = .cache

; OBLIGATOIRE permet d'activer le plugin `tdp_vars`, qui construit les variables utilisées par les collections Ansible de TDP.
[tdp]
vars = tdp_vars

; OBLIGATOIRE permet d'activer le passage à l'utilisateur `root` pour toutes les tâches Ansible.
[privilege_escalation]
become=True
become_user=root
```

Se référer à la [documentation officielle](https://docs.ansible.com/ansible/latest/reference_appendices/config.html#the-configuration-file) pour plus d'informations sur les options de configuration et la localisation du fichier `ansible.cfg`.

Les composants de TDP sont répartis dans différentes collections Ansible. Se référer à la page [Collection](#) pour obtenir la répartition des composants dans les collections.

```yml
---
collections:
  - name: ansible.posix
    version: 1.5.4

  # Obligatoire
  # Contient les services principaux (HDFS, YARN, Hive, etc.) ainsi que les plugins nécessaires à la gestion des variables.
  - name: community.general
    version: 7.1.0
  - name: https://github.com/TOSIT-IO/tdp-collection
    type: git
    version: master
  # Optionnelle
  # Contient des composants supplémentaires (Kafka, Hue, Airflow, etc.).
  - name: https://github.com/TOSIT-IO/tdp-collection-extras
    type: git
    version: master
  # Optionnelle
  # Contient des composants dédiés à l'observabilité (Prometheus, Grafana, etc.).
  - name: community.grafana
    version: 1.5.4
  - name: https://github.com/TOSIT-IO/tdp-observability
    type: git
    version: main
  # Optionnelle
  # Contient des rôles permettant d'installer les prérequis sur des machines (CentOS 7 ou Rocky 8).
  - name: community.postgresql
    version: 2.4.2
  - name: community.crypto
    version: 2.14.0
  - name: https://github.com/TOSIT-IO/tdp-collection-prerequisites
    type: git
    version: master
```

L'installation des collections s'effectue via `ansible-galaxy`. Les collections sont installées dans le premier répertoire spécifié par la propriété `collections_paths` du fichier `ansible.cfg` :

```bash
ansible-galaxy install -r requirements.yml
```

### Dépendances supplémentaires pour TDP Observability (facultatif)

Si TDP Observability a été installé, il est nécessaire d'installer des dépendances supplémentaires :

```bash
pip install -r ansible_dependencies/ansible_collections/tosit/tdp_observability/requirements.txt
```

## Définition de l'inventaire Ansible

L'inventaire Ansible permet de définir les hôtes sur lesquels déployer TDP, la répartition des composants sur ces hôtes et l'utilsation d'éventuels plugins. Pour TDP, l'inventaire est organisé de la manière suivante :

```txt
/path/to/tdp/inventory/
├── group_vars/
│   └── all.yml
├── hosts
├── topologies/
│   ├── 01_tdp
│   ├── tdp_extra
│   ├── tdp_observability
│   └── tdp_prerequisites
└── tdp_vars.yml
```

### Installation du plugin `tdp_vars`

Le plugin `tdp_vars` permet de construire les variables utilisées par les collections Ansible de TDP. Il est à installer en créant le fichier `/path/to/tdp/inventory/tdp_vars.yml` suivant :

```yml
---
plugin: tosit.tdp.tdp_vars
```

### Variables globales

Certaines variables **doivent** être communes à tous les hôtes. Elles sont à spécifier dans le fichier `group_vars/all.yml`.

Créer le fichier `/path/to/tdp/inventory/group_vars/all.yml` en complètant les valeurs suivantes :

```yml
---
domain: # Domaine DNS du cluster
realm: # Nom du royaume Kerberos
kerberos_admin_principal: # Identifiant de l'administrateur Kerberos
kerberos_admin_password: # Mot de passe de l'administrateur Kerberos
kadmin_principal: # Identifiant de l'administrateur Kerberos
kadmin_password: # Mot de passe de l'administrateur Kerberos
```

**Par exemple** :

```yml
---
domain: tdp
realm: TDP.LOCAL
kerberos_admin_principal: 'admin/admin@{{ realm }}'
kerberos_admin_password: admin123
kadmin_principal: '{{ kerberos_admin_principal }}'
kadmin_password: '{{ kerberos_admin_password }}'
```

### Hôtes

Le fichier `hosts` permet de définir les hôtes sur lesquels déployer TDP. C'est un fichier d'[inventaire Ansible](https://docs.ansible.com/ansible/latest/inventory_guide/intro_inventory.html).

Le fichier d'inventaire doit définir au minimum les groupes suivants :

- `edge`
- `master`
- `master1`
- `master2`
- `master3`
- `worker`

Créer un fichier `/path/to/tdp/inventory/hosts` adapté à votre infrastructure en complétant le contenu suivant :

```ini
[edge]
; Liste des hôtes du groupe edge

[master1]
; Liste des hôtes du groupe master1

[master2]
; Liste des hôtes du groupe master2

[master3]
; Liste des hôtes du groupe master3

[master:children]
master1
master2
master3

[worker]
; Liste des hôtes du groupe worker
```

**Par exemple**, pour un cluster de 6 machines (1 edge, 3 masters et 2 workers), le fichier `hosts` sera le suivant :

```ini
[edge]
edge-01 edge01.tdp.local

[master1]
master-01 master01.tdp.local

[master2]
master-02 master02.tdp.local

[master3]
master-03 master03.tdp.local

[master:children]
master1
master2
master3

[worker]
worker-01 worker01.tdp.local
worker-02 worker02.tdp.local
```

### Topologie

Les fichiers de topologies permettent de définir l'organisation des composants (HDFS NameNode, Hive Metastore, etc.) sur les différents hôtes. Ce sont également des fichiers d'inventaire Ansible.

Chaque collection est fournie avec un fichier de topologie d'exemple (fichier `topology.ini`).

`tdp-collection` est la collection principale. Son fichier de topologie doit être lu en premier. Pour cela, il est conseillé de préfixer son nom par `01_`.

Copier le fichier de topologie de `tdp-collection` dans le répertoire `/path/to/tdp/inventory/topologies` :

```bash
cp /path/to/tdp/collections/ansible_collections/tosit/tdp/topology.ini /path/to/tdp/inventory/topologies/01_tdp
```

Si d'autres collections sont installées, il faut également ajouter leurs topologies afin qu'elles soient prise en compte. Les topologies des autres collections peuvent s'appuyer sur les noms des groupes définis dans la topologie de `tdp-collection`.

```bash
cp /path/to/tdp/collections/ansible_collections/tosit/tdp_extra/topology.ini /path/to/tdp/inventory/topologies/tdp_extra
cp /path/to/tdp/collections/ansible_collections/tosit/tdp_observability/topology.ini /path/to/tdp/inventory/topologies/tdp_observability
cp /path/to/tdp/collections/ansible_collections/tosit/tdp_prerequisites/topology.ini /path/to/tdp/inventory/topologies/tdp_prerequisites
```

Ces fichiers sont à éditer en fonction de la répartition des composants souhaitée. Pour désactiver un composant, il suffit de le retirer des topologies.
